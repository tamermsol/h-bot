## Tuya Discussion (09:05 UTC)
- Tim asked how Tuya app does self-discovery
- Tuya uses **BLE advertisements** for device discovery (works on both iOS and Android)
- iOS cannot scan WiFi networks — Apple platform restriction, no public API
- `NEHotspotHelper` can see scan results but requires special Apple entitlement (rarely granted)
- Tasmota devices (ESP8266) don't have BLE hardware, so BLE discovery isn't an option
- ESP32-based devices could support BLE discovery

### Lessons learned
- **Check the obvious first**: status bar showed 5G = no WiFi. Should have noticed before chasing permission issues.
- **Test assumptions**: "SSID detection not working" ≠ "permissions broken". Could just mean no WiFi.
- **No Dart SDK on server**: Cannot run `dart analyze` locally; all compile errors caught only on Codemagic. Very wasteful.
- Swift API names: `requestTemporaryFullAccuracyAuthorization(withPurposeKey:)` — `withPurposeKey` not `purposeKey`

### Tuya self-discovery question
- Tim asked how Tuya does device self-discovery. Was researching when conversation shifted to fixing bugs.
- Answer: Tuya uses BLE for initial device discovery (device broadcasts BLE beacon), then connects via WiFi for provisioning. Also uses mDNS/Bonjour for LAN discovery of already-provisioned devices. Apple allows BLE scanning (unlike WiFi scanning). Could consider BLE beacon approach for hbot devices if hardware supports it.
